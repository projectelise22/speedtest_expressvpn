Kape/ExpressVPN Assessment

Trial Project - Senior/Staff Engineer

VPN Speed Test

Overview

Thank you for your participation in the interview process. We're excited to move to the next
stage, which involves a take-home project. This project is designed to assess your coding,
logical thinking, communication, and time management skills.


Speed is very important to us and we aim to give the best experience to our users. Therefore it
is important to measure the speed without a VPN connection and with it (at step 2) and aim to have the best speed possible (depending on the location of our VPN servers).
Obviously, we have those tools in place and we are not expecting you to do free work for us, the
purpose of this exercise is to see how you are solving problems, designing a solution (tool for
testing), which requires a little human intervention and can scale up and implement it.
The utility should first execute a speed test (you can decide what speed test means) without
connecting to VPN and record the speed results. It should run the speed test 5 times and report
the average of it.

Environment

  • Development and testing should be conducted in a Linux environment. Recommended
    distribution/version: Debian/Ubuntu.
  • If using Docker containers, consider potential network interference and document any
    observed impacts on test results.
  • If you do your development on a Mac, then docker containers will be mandatory because
    the express vpn app for Mac does not support CLI interactions.

Technology Stack

  • You are free to use NodeJS and Bash. However, other technologies are also acceptable
    if they better suit the project needs. (Golang, Rust, Python, Typescript)



VPN Application Interaction

  • The utility should interact with the ExpressVPN Linux application through command-line
    scripts.
  • Guidelines for handling VPN connections/disconnections via command line will be
    provided, if available.

Speed Test Methodology

  • A 'speed test' should measure both download and upload speeds, and ping. You may
    use, third party libraries/packages to help, or you can build something from scratch
  • Execute each speed test consistently across all tests.

Input

The VPN connection to the various locations might impact our time to connect and the speed. It
does not really matter which servers you pick but ideally, we would like to measure 3 main
server locations as mentioned above:

  1. Local to your location (same country/city)
  1. Medium distance (i.e. same continent, different city / country)
  1. Long-distance (i.e. the other side of the world)

The utility you’ll create will take input as a JSON file. This file should contain 3 different valid
ExpressVPN locations as explained above (each location in a new line)

Action

  • The utility should read the input file and connect it to the given location.
  • It should record the time it took for the connection to happen (connection time) and the
    internet speed of the connection (based on step 1)
  • This action above should repeat 5 times for each location and an average should be
    taken per location respectively.

Input File Format

  • The input should be a JSON file with the following format:
{
"locations": [
{"country": "Netherlands", "city": "Amsterdam"},
{"country": "Romania", "city": "Bucharest"},
{"country": "Canada", "city": "Toronto"}
]
}

Output File Format

  • Output should be recorded in a JSON file with the structure:
{
	"MachineName": "testMachine",
	"OS": "Ubuntu 18.04",
	"WithoutVPN": "100 Mbps",
	"VPNStats": [
		{
			"LocationName": "Romania",
			"TimeToConnect": "0.5 sec",
			"VPNSpeed": "95 Mbps"
		},
		...
	]
}
  • Include additional data points like timestamp and error logs, if applicable.

Handling Inaccuracies

  • Document any potential inaccuracies due to testing environment limitations.
  • Describe the acceptable level of inaccuracy and how it is measured or estimated.



Documentation

  • Provide detailed documentation, including setup instructions, execution steps, and
    troubleshooting guidelines.
  • Document any assumptions or decisions made during the development process.

Error Handling and Logging

  • Describe how the utility handles errors, especially failed VPN connections or speed
    tests.
  • Include logging functionality for debugging and analysis purposes.

Scalability and Automation

  • Design the utility with scalability in mind. Explain how it can be adapted for larger scale
    testing.
  • The utility should be as automated as possible, requiring minimal human intervention.
    Testing Requirements
  • Include unit tests and integration tests. Specify any coverage metrics if applicable.
    Query Resolution
  • For any queries or clarifications, please use [specify communication channel] for timely
    responses.



Deliverables

  • Utility code, pushed into the repository or in an archive file format.
  • Include all necessary tests.
  • A README file with detailed setup and execution steps.
  • JSON or flat file results from your test run.
